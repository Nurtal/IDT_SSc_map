# Reviewer 3 — Computational reproducibility, FAIR data, software

**Manuscript**: SSc-MIM: A Curated, SBGN-Compliant Molecular
Interaction Map of Skin Fibrosis in Diffuse Cutaneous Systemic
Sclerosis
**Author**: N. Foulquier
**Target journal**: npj Systems Biology and Applications
**Reviewer expertise**: Reproducible research, FAIR principles,
SBML/MIRIAM standards, scientific software engineering, CI/CD,
container-based research.

---

## Overall recommendation

**Minor-to-major revision.** From a software / reproducibility
perspective this is among the better-engineered submissions I have
seen at npj-SBA. The repository has a coherent layout, dual
licensing, pinned `environment.yml`, three CI workflows, libSBML
validation, `make auto` end-to-end target, and journaling of the
build process. There are real gaps — no Docker / Apptainer image,
no DOI-pinned dependency on the input data, no automated
regeneration of the figures from CI artefacts — but these are
addressable without re-doing the science. I lean toward "minor
revision" on this axis alone; the editor should weigh this against
R1/R2.

## What is already strong

- **Dual licence**: CC-BY 4.0 for the curated map content, MIT for
  the code. Correctly delineated in the LICENSE file.
- **Version control discipline**: 30+ commits in the git log,
  well-structured commit messages, clear phase tagging (`Phase 4c`,
  `feat(phase4c)`, etc.).
- **CI**: three GitHub Actions workflows
  (`validate_sbml.yml`, `lint`, `scripts-smoke`). The validator
  installs `python-libsbml==5.20.4` and runs on every push touching
  `curation/celldesigner/**.xml`.
- **Linters**: `check_module_specs.py` and `check_bib.py` catch
  real curation issues, with documented examples in `journal.md`.
- **Self-documenting Makefile** with `help` target.
- **Reproducibility journal**: `journal.md` is unusually thorough
  and would be a model for other systems-biology projects.

These elements substantially exceed the npj-SBA baseline.

## Major points

### M1. No container image

`environment.yml` pins versions but Conda environment resolution
drifts over time (especially for `scanpy`, `anndata`, `igraph`).
For a paper that will be re-run in 3–5 years, a **Docker or
Apptainer image** built once at release time and pushed to
Quay/Docker Hub (with the image digest cited in the paper) is the
community norm. Please:

- Add a `Dockerfile` based on `mambaorg/micromamba` that materialises
  `environment.yml` and copies the repo.
- Push to ghcr.io (free for public repos) on the `v1.0` tag via a
  GitHub Action.
- Cite the image SHA-256 digest in the Methods section.

Reviewers can then `docker run` the image and reproduce
end-to-end.

### M2. Input data are not DOI-pinned

The four GEO accessions are stable identifiers, but the manuscript
does not specify which **download date** or which **file checksum**
was used. GEO occasionally updates supplementary files
(re-processed BAM, corrected metadata). Please:

- Record SHA-256 of every downloaded archive in
  `analysis/overlay/build_overlay.report.json` (it already exists
  — extend the schema).
- Mirror the four datasets to a Zenodo deposit (one-off, ~5 GB
  total — within the 50 GB free tier) and cite the Zenodo DOI in
  the paper alongside the GEO accession. This insulates the work
  from upstream changes.

### M3. Figures are not regenerated by CI

The PNG / SVG figures live in `figures/` and are committed to the
repository. There is no CI workflow that regenerates them from the
underlying analysis outputs, so they can silently drift from the
data. Please:

- Add a `make figures` target (probably already exists; please
  surface in CI).
- Add a `figures-check` CI job that runs `make figures` and then
  diffs against the committed figures (allow a small tolerance via
  perceptual hash or SHA-256 equality for SVG).
- Alternatively, generate figures into an `.figures/` ignored
  directory in CI and upload as a release artefact, removing the
  binary blobs from the main branch.

### M4. SBML — MIRIAM and BioModels readiness

For npj-SBA the SBML should be **MIRIAM-compliant** so that it can
be auto-deposited at BioModels with no further curation. Current
state:

- `<notes>` blocks contain free-text annotation.
- Species are annotated via TSV side tables, not via the standard
  SBML `<annotation>` RDF blocks.

Please add a step in the build pipeline that emits a
**MIRIAM-annotated** copy of the integrated XML (URIs in the form
`urn:miriam:hgnc:1234`, `urn:miriam:pubmed:12345678`,
`urn:miriam:obo.eco:ECO_0000314`) for BioModels submission. This is
also a prerequisite for the "MINERVA at publication time" claim
flagged by R1.

### M5. SBGN PD round-trip

CellDesigner exports SBML L2v4 with CellDesigner extensions; not
all SBGN-PD diagrams round-trip cleanly through other tools (e.g.
Newt, SBGN-ED, Cytoscape SBGN-PD). For a community resource the
authors should:

- Test that the integrated XML imports cleanly into at least one
  alternative SBGN-PD viewer (Newt or VANTED) and into MINERVA.
- Document any glyph-loss or layout-loss in
  `minerva/deployment_notes.md`.
- Optionally export a clean SBGN-ML file
  (`SSc_MIM_integrated.sbgnml`) as a release artefact.

### M6. Provenance graph

The provenance from "raw Reactome pathway" → "harmonised XML" →
"integrated XML" → "preflight-passed" → "v1.0 release" is recorded
in commits + reports but is not visible to a downstream consumer.
A **PROV-O** or **RO-Crate** manifest at the repo root would make
the data lineage machine-readable. This is becoming standard for
FAIR data and the Disease Maps Project will likely require it
within 2–3 years.

## Minor points

### m1. README pre-1.0 vs current state

The README still describes the project as if Phase 2 curation
were ongoing, with "Phase 2 — Module curation (weeks 4–11)"
language and a "go / no-go week 7" gate that has passed. Please
refresh README to reflect the actual Phase 4c-complete state and
move the original timeline to `docs/historical_roadmap.md` if it
should be preserved.

### m2. `pip install -e .` package metadata

The repository has no `pyproject.toml` and no `setup.py`. For
users to `import ssc_mim` and reuse the scripts as a library,
please add minimal `pyproject.toml` packaging. This is also
needed for tools like Renku, snakemake-workflows, and
nf-core-style distribution.

### m3. Test coverage of scripts

`scripts/` has 16 Python files. There is no `pytest` suite.
Several scripts are non-trivial (e.g. `wire_ssc_tier1.py`,
`integrate_modules.py`, `network_analysis.py`) and merit unit
tests around the SBML round-trip and the HGNC normalisation. A
1-day testing sprint would dramatically de-risk future
maintenance. `pytest` plus a few golden-file tests would be
sufficient.

### m4. Citation of dependencies

`scanpy`, `libSBML`, `NetworkX`, `pandas`, `matplotlib`, `anndata`
should each be cited in the manuscript with the appropriate
journal references. The current Methods §2.9 is a flat list; a
table with version, citation, and use would be more rigorous.

### m5. `.zenodo.json` co-author placeholders

The current `.zenodo.json` still contains REPLACE_ME entries
(per STATUS.md). Please complete before resubmission so that the
DOI mint will carry full metadata. This will also bear on the
ORCID-driven contribution attribution.

### m6. `CITATION.cff` validation

Please run `cffconvert --validate` and `cffconvert -f bibtex` to
make sure the CFF file converts cleanly. The README cites the
JOSS-style "How to cite" but does not include the converted
BibTeX block.

### m7. Logger/debug noise

Several scripts in `scripts/` use `print()` rather than the
standard `logging` module. Migrate to `logging` with a
`-v/--verbose` flag for downstream users who pipe outputs.

### m8. Data version of GSE195452

GSE195452 was updated in 2023 to include corrected metadata. The
journal.md does not record which version was used. Please add a
note specifying the GEO file timestamp.

### m9. License of derived data

If you are publishing `analysis/overlay/cluster_deg_multi.tsv`
which is derived from the four GEO datasets, please confirm that
the derivative-product license is compatible with the source GEO
ToS (it is, but a sentence in DATA.md would be clearer).

### m10. `make auto` total runtime

Please document on a reference machine (e.g. M2 MacBook Pro / 16 GB
RAM, or a Linux laptop) the wall-clock time of `make auto`. Users
considering reproduction will want to know whether this is a
1-hour or a 1-day operation.

## Summary of essential revisions (R3)

1. Provide a container image and cite the digest (M1).
2. Mirror inputs to Zenodo, record checksums (M2).
3. Add CI for figures, prevent silent figure drift (M3).
4. Emit a MIRIAM-annotated SBML for BioModels deposit (M4).
5. Test the integrated XML in at least one alternative SBGN viewer;
   export SBGN-ML (M5).
6. Add an RO-Crate or PROV-O manifest (M6).
7. Refresh README to reflect Phase 4c-complete state (m1).
8. Add minimal `pyproject.toml` and a small `pytest` suite (m2, m3).

These are mostly engineering tasks with negligible risk and high
durability value.
